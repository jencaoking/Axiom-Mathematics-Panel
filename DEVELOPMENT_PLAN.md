# MathLab 2.0 开发计划 (代号: speed)

```speed
## MathLab 2.0 开发计划（中文）

### 总体概述
**MathLab 2.0** 项目旨在将现有的 MathLab 应用升级为现代化、模块化、性能卓越的数学运算与可视化平台。计划围绕三大核心方向展开：

1. **核心架构重构** – 实现异步、线程安全、插件化的底层框架。  
2. **可视化能力提升** – 基于 Three.js + PyQtWebEngine 的 3D 渲染以及 ECharts 的高级图表插件。  
3. **开发者体验** – 完善日志系统、CI/CD 流水线，以及清晰的插件 API，提升后续扩展效率。

计划分为 **四个阶段**（每阶段约 0.5 年），每阶段设定明确的里程碑与交付物。

---

## 阶段 1 – 基础设施（第 1‑3 个月）
| 里程碑 | 描述 | 负责人 | 成功标准 |
|---|---|---|---|
| **线程安全核心** | 将 `AIRequestWorker` 改为继承 `QThread`，把 UI 阻塞的调用迁移到后台线程。 | 核心团队 | 无 UI 卡顿；跨线程 UI 更新错误消除。 |
| **Markdown 清理工具** | 在 `ai_manager.py` 中加入 `clean_markdown`，去除 LLM 响应的代码块与语言标签，确保 JSON 解析有效。 | AI 集成组 | 单元测试中 100% 解析成功。 |
| **插件系统** | 定义通用 `Plugin` 基类，提供 `activate`、`deactivate` 生命周期钩子。 | 架构组 | 可通过配置加载新插件，无需修改代码。 |
| **CI 搭建** | 使用 GitHub Actions 实现代码检查、单元测试、覆盖率报告。 | DevOps | 所有 PR 均通过 CI。 |

---

## 阶段 2 – 可视化引擎（第 4‑6 个月）
| 里程碑 | 描述 | 负责人 | 成功标准 |
|---|---|---|---|
| **3D 插件** | 实现 `plugin_3d_viewer`，使用 Three.js + PyQtWebEngine；通过 `QWebChannel` 将 Python 计算的网格数据传递给前端。 | 可视化团队 | 用户可交互旋转、缩放 3D 曲面。 |
| **ECharts 插件** | 集成 ECharts，实现 2D 图表的双向事件交互。 | 可视化团队 | 实时数据更新能即时反映在图表上。 |
| **性能审计** | 使用 Chrome DevTools `lighthouse_audit` 对渲染帧率、内存使用进行基准测试。 | QA | 常规网格渲染帧率 ≥ 60 FPS，内存泄漏 < 5 MB/小时。 |

---

## 阶段 3 – 高级功能（第 7‑9 个月）
| 里程碑 | 描述 | 负责人 | 成功标准 |
|---|---|---|---|
| **实时协作** | 基于 WebSocket 实现会话共享，支持多人协同操作同一绘图。 | 协作组 | 两位用户可同步查看与操作同一图形。 |
| **插件市场** | 设计插件目录 UI，实现一键发现、安装、更新插件。 | UX 组 | 用户可以“一键”安装新插件。 |
| **自动部署** | 扩展 CI，在标签发布时使用 PyInstaller 打包独立可执行文件。 | DevOps | 每次 `v*` 标签生成发布产物。 |

---

## 阶段 4 – 打磨与发布（第 10‑12 个月）
| 里程碑 | 描述 | 负责人 | 成功标准 |
|---|---|---|---|
| **全局日志系统** | 实现统一日志路由，支持日志等级、文件轮转、UI 日志查看器。 | 日志组 | 错误捕获带堆栈，日志可搜索、可视化。 |
| **内存管理** | 在插件 `on_deactivate` 中调用 `deleteLater()`，确保 Chromium 进程彻底关闭。 | 核心团队 | 插件停用后无残留 Chromium 进程。 |
| **文档与教程** | 完成开发者文档、API 参考与用户教程。 | 文档组 | 文档通过 CI 检查，覆盖率 > 80%。 |
| **公开 Beta** | 向早期用户发布 MathLab 2.0 Beta，收集反馈。 | 产品组 | 至少 20 名 Beta 用户，NPS > 30。 |

---

### 风险与对策
| 风险 | 影响 | 对策 |
|---|---|---|
| **线程安全问题** | 崩溃、UI 卡顿 | 采用 `QThread` 模式统一，编写充分的单元测试。 |
| **WebEngine 内存泄漏** | 长时间使用后 OOM | 在 `on_deactivate` 强制 `deleteLater()`，并使用 `take_heapsnapshot` 定期检查堆快照。 |
| **插件兼容性** | 第三方插件破坏 | 采用版本化插件 API，提供兼容性警告。 |
| **性能回退** | 渲染卡顿 | 引入 Lighthouse CI 指标，使用 `performance_start_trace` 进行性能分析。 |

---

### 立即可执行的行动建议
1. **完成线程安全核心** – 合并已改造的 `AIRequestWorker`，并补充单元测试。 
2. **实现 Markdown 清理工具** – 将其集成到 LLM 响应流水线。 
3. **搭建 CI 流水线** – 确保代码检查、单元测试与覆盖率在每次提交后自动运行。 
4. **编写插件规范文档** – 明确插件必须实现的钩子与数据交互格式。 

以上工作为后续的可视化插件奠定坚实基础。
```
